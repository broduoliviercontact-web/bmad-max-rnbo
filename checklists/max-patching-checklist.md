# Max Patching Checklist

- [ ] Le patch a un module d'entrée clair.
- [ ] Le patch a un module de sortie clair.
- [ ] Les sous-patchers sont nommés.
- [ ] Les send/receive sont documentés.
- [ ] Les fan-out critiques utilisent `trigger`.
- [ ] Les valeurs par défaut sont définies.
- [ ] Les commentaires expliquent les modules.
- [ ] Le patch peut être testé par sections.
- [ ] Les dépendances sont dans le projet.
- [ ] Les chemins absolus sont évités.
